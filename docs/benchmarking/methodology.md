# Benchmark methodology

Alpha.4 comparison separates raw jsQR, raw ZXing-JS, raw ZXing-C++ WASM, each Scanly profile, three single-engine ablations, and sequential/experimental-parallel JS+WASM orchestration. Cold initialization, a second warm initialization call, decode latency, attempts, controlled memory, WASM linear-memory peak, installed footprint and unique wins are labelled separately. Source identity includes the binary, native-boundary and loader hashes.

The canonical internal suite has three visible partitions: deterministic generated transformations, project-owned real images, and deterministic negative/adversarial images. The separate `externalOpenLicenseRealWorld` cohort contains third-party Wikimedia Commons photographs and is never counted as project-owned evidence. `benchmark/datasets/blind/` defines an intentionally empty separation boundary for a future blind set that must not be used for tuning.

External open-license photographs provide third-party real-world validation but do not satisfy the project-owned photograph release gate.

All benchmark modes measure the same Node SDK path: `NormalizedFrame -> CaptureRouter -> compiled graph -> registered engines`. `npm run benchmark` produces ignored development evidence; `benchmark:canonical-candidate` produces one clean candidate profile; `benchmark:assemble-canonical` verifies the four-report evidence set before `benchmark:update-canonical` can install tracked aliases. Full Benchmark produces transient CI artifacts in independent jobs. A canonical-compatible run performs one cold initialization, at least one excluded warmup, and at least three measured iterations per fixture. Every required measured iteration must pass. Reports retain bounded run timings, pass/failure counts, and unstable-payload evidence; median latency is representative only after correctness is established.

Variance terminology is strict: `perFixtureRunStdDevMs` summarizes repeated timings of the same fixture, while `fixtureLatencySpreadMs` describes differences among fixtures (standard deviation and median absolute deviation). Cross-fixture spread is never called run-to-run variance. Machine latency remains in detailed reports and is not a universal product claim.

`npm run benchmark:profiles` independently gates fast, balanced, and robust against immutable profile-specific baselines selected by `benchmark-results/baselines/registry.json`. Timing comparison is allowed only when runtime, OS, architecture, Node major version, scenario, and dataset identity are compatible. Smoke runs on a different CI environment enforce functional gates without treating Windows and Linux timing as interchangeable. Profile-specific relative and absolute limits are encoded in the quality gate. Recall, exact payload, false positives, multi-code completeness, attempts, timeout, cancellation, and engine failures have direct gates.

`npm run benchmark:compare` sends the same decoded RGBA buffer to raw jsQR, raw ZXing JavaScript, and Scanly fast/balanced/robust. Installed dependency footprint is labeled `installedPackageFootprintBytes`; it is not browser bundle size. `npm run bundle:analyze` separately reports esbuild unminified, minified, gzip, Brotli, Worker, engine, and tree-shaken entry costs. Runtime and bundle metrics are never mixed.

Parallel engine execution shares one global deadline and atomic attempt budget. The primary branch may use the full budget while all secondary initial allocations are bounded to 30%, preserving at least 70% primary opportunity and allowing the primary to borrow unused attempts. Single-code lower-priority branches are cancelled after a valid higher-priority result; multi-code branches finish and merge spatially. The current development comparison remains 1/63 below the 1% recall/accuracy parity tolerance, so parallel is explicitly experimental and every built-in production scenario remains sequential.

Canonical and baseline handling follows the [artifact-driven evidence lifecycle](evidence-lifecycle.md). Candidate generation never overwrites tracked aliases, and baseline freeze consumes the assembled external manifest rather than tracked comparison output.

The suite is internal regression evidence. It is not a universal accuracy figure, an ML evaluation, or a commercial competitor comparison. The retained `14-damaged` failure remains in the denominator. Commercial results require licensing and methodology approval described in `benchmark/competitors/README.md`.
